[52] Section 1 - Requirement Breakdown
[53] The Notes app lets users create, edit, and delete notes across three categories: Home, Work, and Personal. Testing covers the full SDLC - UI interactions, REST API calls, and the data sync between them. The table below lists every functional and non-functional requirement this plan addresses.
[54] 
[55] Req ID
[56] Module
[57] Description
[58] FR-01
[59] Authentication
[60] User should be able to log in with valid credentials via UI
[61] FR-02
[62] Note Creation
[63] User can create a note with Category, Title, and Description via UI
[64] FR-03
[65] Note Visibility
[66] Newly created note must appear immediately in the UI list without page refresh
[67] FR-04
[68] API – List Notes
[69] GET /notes endpoint must return a well-formed list of all notes for the logged-in user

[76] FR-07
[77] Delete Sync
[78] Once a note is deleted via API, it must no longer appear in the UI list
[79] FR-08
[80] API Performance
[81] All API responses must complete within 2 seconds under normal load
[82] FR-09
[83] Negative Validation
[84] System must handle invalid inputs and unauthorized access with appropriate error responses

[121] Section 2 - Manual Test Plan
[122] 2.1 Purpose
[123] This plan covers baseline manual validation of the Notes app before automation kicks in. The goal is to confirm that UI flows work as expected, the API behaves correctly, and both layers stay in sync with each other. Passing these manually gives us confidence the app is stable enough to automate.
[124] 2.2 Scope
[125] In Scope
[126] UI: user registration, login, note creation, note edit, note deletion, category filtering
[127] API: POST /users/login, GET /notes, POST /notes, PATCH /notes/:id, DELETE /notes/:id
[128] End-to-end data sync: UI-created notes verified in API, API-deleted notes verified absent in UI
[129] Negative cases: wrong credentials, missing fields, unauthenticated API calls
[130] 
[131] Out of Scope
[132] Performance load testing (handled in Section 3)
[133] Security penetration testing
[134] Cross-browser compatibility (primary browser: Chrome)
[135] 
[136] 2.3 Test Strategy
[137] 
[138] Layer
[139] Approach
[140] Tools
[141] UI Testing
[142] Structured test cases + exploratory runs
[143] Chrome Browser, Selenium (manual run)
[144] API Testing
[145] Request/response validation, schema checks
[146] Postman, Swagger UI
[147] E2E / Hybrid
[148] Create via UI → verify via API, delete via API → verify UI
[149] Manual cross-check

[151] Boundary, invalid data, unauthorized access
[152] Both UI and API
